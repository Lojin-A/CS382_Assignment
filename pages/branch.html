⚠️ Merge conflicts

If two people edit the same file, Git needs your help to choose which changes to keep. Conflicts must be resolved manually before merging.

Why use Pull Requests?

  • Code review catches bugs before they reach main
  • Documentation of changes and decisions
  • Team awareness of what others are working on
  • Quality control gatekeeper

Best Practices

Do's and Don'ts of branching

✓ Do This

  • Use descriptive branch names
  • Keep branches short-lived
  • Sync with main regularly
  • Delete merged branches
  • Write clear commit messages

✗ Avoid This

  • Commit directly to main
  • Vague names (test, temp, new)
  • Long-running feature branches
  • Merge without review
  • Forget to pull latest changes
# Branch naming convention examples: feature/login-page feature/shopping-cart bugfix/header-error hotfix/security-patch docs/update-readme

Complete Branching Workflow

From start to finish

1 git checkout -b feature/new Create and switch to branch
2 git add . && git commit -m "message" Stage and commit changes
3 git push origin feature/new Push to GitHub
4 Create Pull Request Review and merge on GitHub
5 git branch -d feature/new Delete branch after merge

Key Takeaway

Always create a new branch for each feature or bug fix. This keeps main stable and enables smooth team collaboration.
